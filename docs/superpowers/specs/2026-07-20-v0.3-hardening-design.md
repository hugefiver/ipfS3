# v0.3 Hardening Design

**Status:** Ready for implementation
**Date:** 2026-07-20
**Scope:** The six `ROADMAP.md` v0.3 items and the confirmed security corrections in this document

## Goal

Complete v0.3 without changing the gateway's IPFS-backed storage model, CID-as-ETag contract, conservative pin policy, or standard S3 routing path. The release includes AWS bucket validation, presigned standard Put/Get coverage, Head Range behavior, CID response headers, real-TCP coverage, and safe SSE-C behavior for standard objects and multipart uploads.

## Non-goals

- No remote pinning provider, CID reference counting, public HTTP IPFS gateway, or encrypted chunk-level Range optimization.
- No storage of a raw SSE-C key, its base64 transport value, or its key-MD5 value.
- No master-key rotation protocol in v0.3.
- No custom route for presigned requests.

## Chosen security approach: C2

Persist a versioned HMAC fingerprint for every new SSE-C object and multipart upload. A row with a stored fingerprint can reject a mismatched customer key in constant time before Kubo access. A pre-migration SSE-C object has a `NULL` fingerprint. Its first valid SSE-C read authenticates the full ciphertext and exact plaintext size, then atomically claims the fingerprint by immutable object ID. A wrong key can never claim the row.

This approach avoids an always-on Kubo read for modern objects while preserving access to existing ciphertext. It is preferable to accepting a key based only on successful decryption, and to a one-time migration that cannot recover customer keys.

## Architecture and boundaries

```text
client -> axum -> s3s -> S3Impl -> s3/ops -> store / crypto / Kubo
```

- `crypto/chunker.rs` generates a fresh random nonce for every newly encrypted chunk.
- `crypto/key.rs` derives and verifies the persisted fingerprint.
- `store/object.rs` and `store/multipart.rs` persist fingerprints and atomically claim legacy rows.
- `s3/ops/object.rs` validates normal and copy-source SSE-C headers, authenticates legacy objects, and sets SSE-C response fields.
- `s3/ops/multipart.rs` authenticates each claimed part before a root add or pin.
- `tests/integration.rs` supplies real-TCP protocol and Kubo side-effect evidence.

## Encryption nonce format and compatibility

Every newly encrypted chunk uses an independently generated CSPRNG 96-bit nonce. `encrypt_chunk_stream` accepts only the input stream and `Arc<ObjectKey>`; it no longer receives `object_id`, `part_number`, or a chunk index. The byte format remains:

```text
nonce (12 bytes) || ciphertext || GCM tag (16 bytes)
```

Existing ciphertext that used an HKDF-derived nonce remains readable because decryption consumes the nonce embedded in each chunk. The HKDF nonce derivation method and all production calls to it are removed after a fixed legacy ciphertext decrypt test is in place.

The random-nonce invariant applies to normal PutObject, multipart UploadPart, and multipart Complete re-encryption. Retrying Complete, replacing a part, or encrypting equal plaintext with the same key must not reuse a nonce.

## Multipart SSE-C verification

SSE-C headers are an indivisible set: algorithm, key, and key-MD5. The algorithm is `AES256`; mixed SSE-S3 and SSE-C headers, missing headers, malformed encoding, invalid length, and wrong key-MD5 return `InvalidArgument`.

`authenticate_sse_c_parts` receives ordered `(cid, expected_plaintext_size)` pairs. Before the first Kubo `cat`, it preflights every recorded size and rejects any negative value. It then decrypts each full immutable CID stream, uses checked accumulation for every yielded plaintext chunk, and requires the observed count to equal the recorded non-negative part size. A recorded size of zero succeeds only when the decrypted plaintext is also empty. Arithmetic overflow, empty ciphertext with a positive expected size, truncated ciphertext, or a mismatch at a full encrypted-chunk boundary returns `InvalidPart`. This pass happens before root `add` or `pin`.

After successful authentication, Complete reads each CID again to stream decrypt, concatenate, and encrypt the root. CID immutability makes that second read the trust boundary. The second read avoids buffering the whole object and does not weaken the pre-authentication ordering.

## SSE-C fingerprint persistence

The gateway stores this domain-separated value:

```text
v1:hmac-sha256:<lowercase hex>
HMAC-SHA256(master_key,
  "ipfs-s3-gateway/sse-c-key-fingerprint/v1" || 0x00 || customer_key_bytes)
```

The same current untracked v0.3 migration adds nullable `sse_c_key_fingerprint TEXT` to both `objects` and `multipart_uploads`; down drops both columns. SQLite migration tests retain object, upload, and part data and the multipart part cascade foreign key. PostgreSQL builder tests prove the migration produces only `ALTER TABLE` statements.

`object::Model`, `LatestObjectRow`, object upsert/write helpers, completion reconciliation, and every `LatestObjectRow` construction carry the field. Plaintext and SSE-S3 rows always store `NULL`. Standard SSE-C PutObject and SSE-C Complete store the fingerprint. CopyObject copies the verified source fingerprint to its new object row.

The store has separate atomic claim helpers for legacy multipart uploads by `upload_id` and legacy objects by immutable `objects.id`. Each performs conditional `UPDATE ... WHERE ... fingerprint IS NULL`, reloads the row, and lets the caller constant-time verify the reloaded winner.

## GetObject and HeadObject C2 behavior

For an SSE-C object, Get and Head require a complete, non-mixed normal SSE-C header set. For a fingerprinted row, the gateway verifies the supplied key before any Kubo access. A well-formed different key returns `AccessDenied` with zero Kubo calls. Header-set failures return `InvalidArgument`.

For a legacy `NULL` fingerprint, the gateway performs one full cat/decrypt pass and checks the exact stored plaintext size. Only that success can claim the fingerprint. It reloads the row and verifies the winner before continuing. A wrong key, malformed ciphertext, or size mismatch does not claim the row.

The first no-Range legacy Get pre-authenticates with one full cat, claims the fingerprint, then performs a second cat for its streaming response. A ranged legacy Get may use its authenticated plaintext to produce the selected range. A first legacy Head or Head Range performs exactly one authentication cat and no response-body read. Once claimed, Head and Head Range return with zero Kubo calls as before.

Successful SSE-C Get and Head set `sse_customer_algorithm` to `AES256` and `sse_customer_key_md5` to the validated request key's base64 MD5. They never expose the customer key.

## CopyObject SSE-C behavior

CopyObject identifies source SSE-C credentials only through these source-specific headers:

```text
x-amz-copy-source-server-side-encryption-customer-algorithm
x-amz-copy-source-server-side-encryption-customer-key
x-amz-copy-source-server-side-encryption-customer-key-md5
```

The gateway uses a parameterized header parser so normal object reads use the normal names and CopyObject uses these names, with identical complete-set, `AES256`, base64, length, and MD5 rules. For a fingerprinted source, a wrong key is `AccessDenied` before Kubo pin or database publication. For a legacy source, C2 full authentication and atomic claim occur before pin or publication. A successful SSE-C copy writes the source fingerprint to the destination `LatestObjectRow`; a wrong key never publishes a destination row.

## Master-key lifetime

The configured `IPFS_S3_MASTER_KEY` is part of SSE-C fingerprint verification. It must remain unchanged for the whole lifecycle of every SSE-C object and multipart upload: Create or Put, UploadPart, Complete, Get, Head, and CopyObject. Changing it can make a valid stored fingerprint unverifiable even though ciphertext still carries its nonce. v0.3 does not support rotation. README configuration guidance must state this operational requirement.

## Existing v0.3 behavior retained

`AwsNameValidation` is explicitly installed in production and the real-TCP harness. Head Range changes only `Content-Length`, has no body or `Content-Range`, and does not touch Kubo for non-legacy rows. Successful standard PutObject returns the CID ETag plus `x-amz-meta-ipfs-cid` and `x-amz-meta-ipfs-url: ipfs://<cid>` only after add, pin, and database publication. Standard presigned requests remain handled by s3s query SigV4.

## Error mapping

- Partial, mixed, malformed, or invalid normal or copy-source SSE-C header sets: `InvalidArgument`.
- A valid but nonmatching SSE-C fingerprint: `AccessDenied`.
- Corrupt, empty, truncated, overflowed, or size-mismatched authenticated multipart data: `InvalidPart` before root add or pin.
- Kubo, database, pinning, and malformed persisted-fingerprint failures retain their existing infrastructure mappings.

The implementation uses TDD. Reviewer scenario A proves nonce uniqueness and legacy decrypt compatibility. Scenario B proves multipart pre-authentication rejects every recorded-size and ciphertext boundary failure before root mutation. Scenario C proves C2 fingerprinted and legacy Get/Head behavior, response fields, error codes, atomic claims, and Kubo counts. CopyObject, every object publication path, multipart completion reconciliation, migration up/down behavior, and normal v0.3 acceptance behavior are covered.

## Invariants

1. CID remains the object ETag.
2. SSE-C raw key material and transport MD5 are never persisted or logged.
3. Each newly encrypted chunk has an independent CSPRNG nonce.
4. All new SSE-C objects and multipart uploads have a fingerprint before publication.
5. A mismatched fingerprint rejects before Kubo access; an unclaimed legacy row is claimed only after full cryptographic and size authentication.
6. Multipart root add and pin occur only after each SSE-C part has passed authenticated exact-size validation.
7. CopyObject uses copy-source, not normal request, SSE-C headers for an encrypted source.
8. The configured master key remains stable for every SSE-C object's lifetime.
9. Conservative pin semantics remain unchanged.
